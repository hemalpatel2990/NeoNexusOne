Modular Level Kit — Approach

How should the kit pieces be created and managed?

A

Blueprint Actor Kit (Recommended)

Each prop is a Blueprint Actor with a Static Mesh Component + per-instance color parameter exposed as an editable property. Built from UE5's basic shapes (Cube, Cylinder, Cone, Sphere) combined in the BP. No external modeling needed.

Pros

  • Per-instance color tinting via a single Blueprint property
  • Can add collision, snap points, and gameplay tags per-prop
  • Easy to extend — add new props by duplicating and reshaping
  • Works with existing M_EchoMaster + Dynamic Material Instance

Cons

  • Each BP is a full Actor (slight overhead vs raw static mesh)
  • Compound shapes (tree = cylinder + sphere) need manual assembly
B

Static Mesh + Material Instance Kit

Each prop is a raw Static Mesh placed directly in the level. Per-instance color via Material Instance Dynamic created at runtime by a lightweight C++ component attached to each mesh.

Pros

  • Minimal overhead — just meshes with a small component
  • Fastest rendering (instanced static meshes)
  • Clean separation: mesh = shape, component = color

Cons

  • Needs a new C++ UActorComponent for per-instance color
  • Less self-contained than BPs — two things to manage per prop
  • Compound shapes harder without a parent Actor
C

C++ Base Actor + BP Children

A single C++ base class (AEchoLevelProp) that handles per-instance color, collision setup, and sonar integration. Each prop is a Blueprint child that just sets its mesh(es) and default color.

Pros

  • Consistent behavior across all props (color, collision, tagging)
  • C++ handles the "smart" part, BPs just configure shape/defaults
  • Easy to add gameplay features later (destructible, interactive)
  • Follows the project's existing C++ base + BP child pattern

Cons

  • More upfront work (C++ class + BP children)
  • Requires editor restart for C++ changes
